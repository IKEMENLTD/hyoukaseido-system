1. システム概要
================================================================================

1-1. 目的
--------
イケメングループ(7事業部)の人事評価を一元管理するWebシステム。
四半期OKR x 半期査定のハイブリッド評価制度をデジタル化する。

1-2. 技術スタック
-----------------
フロントエンド : Next.js 16 (App Router) + TypeScript + React 19
スタイル       : Tailwind CSS v4 (INDUSTRIAL VOIDデザインシステム)
バックエンド   : Supabase (PostgreSQL + Auth + RLS)
認証           : Supabase Auth (Google OAuth)
通知(組織)     : Slack Webhook / LINE Messaging API / ChatWork REST API (トークンはDB管理)
通知(個人DM)   : OAuth連携 (Slack/LINE/ChatWork) ※未設定
ホスティング   : Netlify (https://hyoukaseido.netlify.app)
主要パッケージ : @supabase/supabase-js, @supabase/ssr, next, react
UIユーティリティ: lucide-react (SVGアイコン), recharts (グラフ), clsx, tailwind-merge, class-variance-authority
設定ファイル   : tsconfig.json, next.config.ts, postcss.config.mjs

1-3. 設計原則
-------------
- Server Component優先: データ取得はサーバーサイドで実行
- Server Actions: 評価系の全データ変更はServer Actions経由(権限チェック+バリデーション)
  ※ OKR系(checkin/objectives作成/review)はクライアント直接操作+RLS依存(意図的設計)
    RLS: objectives INSERT=本人orG3+, checkins INSERT=本人のみ, key_results=objective所有者orG3+
    クライアント操作はリアルタイムUI更新と実装シンプルさを優先。権限はRLSで十分保護済み
- RLS二重防御: アプリ層+DB層の両方で権限制御
- 楽観ロック: ステータス遷移は .eq('status', currentStatus).select().single() で検証
- 型安全: any禁止、全Supabaseクエリに手動型定義
- 絵文字禁止: SVGアイコンのみ使用 (lucide-react)

1-5. INDUSTRIAL VOIDデザインシステム
------------------------------------
背景: #050505 / パネル: #0a0a0a / ボーダー: #1a1a1a
テキスト: #e5e5e5 / サブ: #737373 / 非活性: #404040
アクセント: #3b82f6 (blue) / 成功: #22d3ee (cyan) / 警告: #f59e0b / エラー: #ef4444
ネオン(1%使用): #ccff00
禁止: 角丸(border-radius)、シャドウ(box-shadow)、暖色グラデーション
SVGアイコン: strokeLinecap="square", strokeLinejoin="miter"

1-4. 権限モデル
--------------
G1 (メンバー)   : 自分のデータのみ閲覧・編集
G2 (シニア)     : G1と同等
G3 (マネージャー): 自事業部のデータ閲覧・上長評価・1on1記録
G4 (事業部長)   : 全事業部データ閲覧・管理機能・キャリブレーション
G5 (代表)       : 全データ・全機能・組織設定変更

6. セキュリティモデル
================================================================================

6-1. 認証
---------
- Google OAuth (Supabase Auth)
- middleware.ts で未認証ユーザーを /login にリダイレクト
- auth/callback でメール自動リンク (email_confirmed_at チェック付き)
- next パラメータは許可リスト方式でオープンリダイレクト防止

6-2. 認可 (アプリ層)
--------------------
- admin/layout.tsx: G4/G5のみ (Server Component一括ガード)
- 各Server Action: verifyEvaluationAccess() で権限・ステータス・事業部チェック
- G3の事業部横断アクセスをブロック (division_ids照合)

6-3. 認可 (DB層 - RLS)
-----------------------
- 全テーブルでRLS有効
- マルチテナント: 全テーブルにorg_idフィルタ適用(自組織のデータのみアクセス可)
  - 直接: divisions, grade_definitions, value_items, okr_periods, eval_periods, crosssell_routes
  - 間接(division_id経由): division_members, kpi_templates
  - 間接(template_id経由): kpi_items, behavior_items
  - notification_channels: org_id + G4/G5等級チェック
- evaluations: member_id/evaluator_id/grade+division_idで階層制御
- eval_*_scores: evaluationsテーブルを経由した間接RLS
- members_select: USING(true) だが salary は G4/G5のみアプリ側で制御
- 管理系テーブル: G4/G5のみ書き込み可(modifyポリシーでorg_id+等級チェック)

6-4. 楽観ロック
--------------
- 全ステータス遷移で .eq('status', currentStatus).select('id').single()
- 0行更新を data=null で検出し、重複操作を防止

6-5. 入力バリデーション
-----------------------
- BehaviorScore: 1-4 範囲チェック
- ValueScore: 1-10 範囲チェック
- テキスト: 1000-5000文字上限
- 数値: NaN/Infinity チェック
- 通知API: レートリミット(last_sent_atベース、DBレベル)、URL相対パスのみ(regex検証)、文字数制限
- OAuth State: HMAC-SHA256署名(OAUTH_STATE_SECRET必須、空文字列フォールバック禁止)
- トークン暗号化: AES-256-GCM (OAUTH_TOKEN_ENCRYPTION_KEY 32バイト検証付き)

7. 評価エンジン
================================================================================

7-1. スコア計算フロー
---------------------
KPI実績入力 -> 達成率算出 -> KPIランク判定 -> 加重平均 -> quantitative_score
行動評価入力 -> 平均(1-4) -> 0-100変換 -> qualitative_score
バリュー評価入力 -> 合計/最大値*100 -> value_score

quantitative_score * weight% + qualitative_score * weight% + value_score * weight%
= total_score (0-100)
-> ランク判定 (rank_thresholds DBカスタム or S:90/A:80/B:60/C:40/D:0)
-> 上位行動ボーナス適用 (+1/+2段階)
-> 推奨昇給額 (rank_thresholds DBカスタム or S:+5万/A:+3万/B:0/C:-2万/D:-5万)
-> 昇格適格性 (S=immediate, 2期連続A=candidate)

7-2. フェーズ別ウェイト
-----------------------
黒字事業部 (profitable): 定量50% / 定性30% / バリュー20%
赤字事業部 (investing) : 定量30% / 定性45% / バリュー25%

判定方法:
  1. division_financials に直近四半期の実績がある → 営業利益>=0 なら profitable
  2. 実績データなし → divisions.phase の手動設定値をフォールバック
  ※ 評価作成時(get-or-create-evaluation.ts)にスナップショットされ、以降変わらない

7-3. ステータス遷移
-------------------
評価期間: planning -> target_setting -> self_eval -> manager_eval -> calibration -> feedback -> closed
個別評価: draft -> self_submitted -> manager_submitted -> calibrated -> feedback_done -> finalized

9-1. 環境変数
-------------
【必須】
NEXT_PUBLIC_SUPABASE_URL       Supabase プロジェクトURL
NEXT_PUBLIC_SUPABASE_ANON_KEY  Supabase anon key
SUPABASE_SERVICE_ROLE_KEY      Supabase service role key (RLSバイパス用)
NEXT_PUBLIC_APP_URL            アプリケーションURL (https://hyoukaseido.netlify.app)

【Cronジョブ用】
CRON_SECRET                    Cron認証トークン

【個人DM通知用 (OAuth連携時のみ)】
CHATWORK_CLIENT_ID             ChatWork OAuthクライアントID
CHATWORK_CLIENT_SECRET         ChatWork OAuthクライアントSecret
OAUTH_STATE_SECRET             OAuth State HMAC署名鍵
OAUTH_TOKEN_ENCRYPTION_KEY     トークン暗号化鍵 (64文字hex = 32バイト)
SLACK_BOT_TOKEN                Slack Bot Token (個人DM用)
LINE_MESSAGING_CHANNEL_ACCESS_TOKEN  LINE チャネルアクセストークン (個人DM用)

※ ChatWork/LINEの組織チャネル通知用APIトークンは環境変数不要
  → 管理画面(admin/notifications)からDB保存

9-2. 静的アセット
-----------------
public/sw.js           Service Worker (オフラインキャッシュ, API/authリクエスト除外)
public/manifest.json   PWAマニフェスト
src/app/globals.css    Tailwind + @media print(.no-printクラスで印刷非表示制御)

9-3. ビルド
-----------
npm run build (Next.js)
Netlify: 自動デプロイ (main ブランチ push)

9-4. DB初期化
-------------
supabase/schema.sql                              テーブル+RLS+インデックス作成
supabase/seed.sql                                初期データ (組織/事業部/等級/KPIテンプレート/経路)
supabase/migration_001_security_and_financials.sql  rank_thresholds + division_financials
supabase/migration_002_oauth_account_links.sql      OAuth連携テーブル + notification_preferences拡張
supabase/migration_003_shared_costs.sql             共有コストテーブル
supabase/migration_004_notification_api_token.sql   api_tokenカラム + RLSポリシー更新
supabase/migration_005_rls_org_id_filter.sql        10テーブルにorg_idフィルタ追加
.env.local.example                                  環境変数テンプレート
